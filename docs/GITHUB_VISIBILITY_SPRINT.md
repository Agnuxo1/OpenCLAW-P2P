# GitHub Visibility Sprint

This document keeps OpenCLAW-P2P visibility work explicit, measurable, and aligned with GitHub community rules. The goal is to make the project easier to understand, easier to try, and easier to cite without mass outreach or repeated unsolicited comments.

## Current anchor

OpenCLAW-P2P is the primary repository for the ecosystem. It should be the public entry point for:

- the protocol narrative
- the Lean 4 verification work
- the live P2PCLAW network
- the MCP server and agent integration path
- the paper trail and reproducibility claims

Satellite repositories should point back here instead of splitting attention across many independent calls to action.

## North-star target

Reach 1,000 GitHub stars through product quality, documentation clarity, reproducible demos, credible research artifacts, and invited community participation.

## Non-spam policy

We will not use automated mass comments, repeated bumps, unrelated issue creation, or unsolicited promotional PRs.

Acceptable visibility work:

- improve our own README, docs, demo, releases, and examples
- submit PRs only where the project clearly accepts the category of contribution
- respond once, politely, when maintainers ask for clarification
- stop contacting repositories that close the discussion as unwanted
- track closed or hostile repositories internally so they are not contacted again

## Sprint checklist

### 1. Improve conversion on this repository

- Move a concise value proposition near the top of the README.
- Add a `Start here` table for researchers, agents, developers, and reviewers.
- Consolidate duplicate Quick Start sections.
- Add a short reproducible demo path that works in under five minutes.
- Add screenshots or a short demo GIF for the live app and agent workflow.
- Add repository topics aligned with actual project scope: `decentralized-science`, `ai-agents`, `mcp`, `lean4`, `formal-verification`, `ipfs`, `peer-review`, `research-tools`.

### 2. Make the ecosystem easier to navigate

- Keep OpenCLAW-P2P as the main public entry point.
- Add clear back-links from CAJAL, p2pclaw-mcp-server, benchclaw, PaperClaw, EnigmAgent, and The-Living-Agent.
- Use consistent short descriptions across repository READMEs.
- Add a simple architecture diagram that shows which repository owns each layer.

### 3. Publish credible artifacts

- Create tagged releases for stable milestones.
- Add a `CITATION.cff` file matching the current arXiv paper.
- Add a `SECURITY.md` and `CONTRIBUTING.md` with maintainer expectations.
- Add reproducibility notes for the Lean proof corpus and live gateway.

### 4. Community participation

- Maintain a small list of genuinely relevant awesome lists and research tooling repositories.
- Open PRs only when the repository guidelines explicitly allow additions and the project satisfies their criteria.
- Do not bump dormant PRs unless a maintainer asks for changes or the PR needs a factual update.
- Thank maintainers when they accept, decline, or explain criteria.

### 5. Measurement

Track weekly:

- stars and forks on OpenCLAW-P2P
- stars flowing to satellite repositories
- accepted PRs in relevant lists
- live site traffic from GitHub
- README click paths to paper, live app, and MCP server
- issues opened by real users

## First milestone

A practical first milestone is 100 stars on OpenCLAW-P2P with a cleaner README, one working demo path, one release, and a small number of accepted, guideline-compliant PRs in relevant repositories.

## Operating rule

Visibility work should leave the project better even if nobody replies. If an action does not improve documentation, reproducibility, usability, or an invited community thread, it should not be done.
